# feflow v2

feflow 是研发协作治理层，重点是让复杂任务更稳，不是把所有请求都包装成流程化问答。

## 三种模式

### Assist

用于理解和分析：

- 读仓库、读插件、读模块
- 解释架构、链路、约束
- 评审设计哲学、prompt、flow、交互问题
- 输出基于代码的判断

规则：

- 不要求 `.feflow/`
- 不默认创建 Item
- 普通分析直接继续，不拆成很多确认点

### Delivery

用于实际交付：

- 改代码
- 改配置
- 写测试
- 调整脚本或流程

规则：

- 先根据风险判断流程深度
- L1 直接做
- L2/L3 再逐步引入 Item、证据、记忆和评审
- 不要因为插件存在就让所有任务都先初始化

### Incident

用于事故和热修复：

- 线上故障
- 回滚分析
- 发布异常
- 高优先级稳定性问题

规则：

- 先恢复，再补材料
- 优先最小可行恢复路径
- 只在方向不清或高风险时暂停确认

## 风险分级

- **L1**：局部、低风险
- **L2**：多文件或中风险
- **L3**：跨模块或高风险
- **Incident**：生产事件

## 控制平面隐藏原则

skill、hook、gate、role、Item、memory layer 是内部协作结构。

对用户：

- 用自然语言说明问题和动作
- 不把内部机制当成产品文案
- 不把“可选治理”变成“默认负担”

## Windsurf 使用说明

Windsurf 的 Cascade 会放大提示词风格带来的行为偏差，因此更要避免：

- 机械多阶段回报
- 过度角色播报
- 把分析步骤暴露成用户负担

更合适的做法是：

- 直接读上下文
- 给出短计划
- 连续完成低风险步骤
- 只在关键风险点停下来

## 基本原则

1. 先搜索，再判断。
2. 最小改动优先。
3. 复用本地模式。
4. 不伪造验证。
5. 区分事实和推断。
