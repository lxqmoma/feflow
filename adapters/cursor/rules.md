# feflow v2

feflow 是前端研发协作治理层，不应该把所有任务都变成“先立项、再分阶段确认”。

## 三种模式

### Assist

用于只读和分析类任务：

- 理解仓库或插件
- 解释架构、模块、数据流
- 评审 prompt / flow / 交互设计
- 分析风险、耦合、可维护性问题

规则：

- 直接读代码和搜索，不默认创建 Item
- 不要求 `.feflow/` 已初始化
- 不要把 skill、hook、gate、role 这些内部术语当成主要输出
- 普通分析不要拆成多轮逐步确认

### Delivery

用于真正要落地的任务：

- 改代码
- 改配置
- 写测试
- 改文档
- 修脚本或工作流

规则：

- 先按风险判断，不默认走重流程
- L1 小改动可以直接做
- L2/L3 再考虑 Item、证据、评审和记忆沉淀
- 只有治理确实有收益时，才要求 `/feflow:init`

### Incident

用于线上事故、紧急回滚、热修复和高优先级故障。

规则：

- 先止血，再补治理材料
- 优先恢复业务和隔离影响
- 只在高风险或方向不明确时停下来确认

## 风险分级

- **L1**：局部、低风险、可快速验证
- **L2**：多文件或中等风险
- **L3**：跨模块、迁移、对外行为有明显影响
- **Incident**：生产事件或时间敏感故障

## Item 策略

只有这些情况才默认创建 Item：

- 任务跨多个会话、多人协作或存在明确依赖
- 需要长期追踪、依赖管理或证据链
- 用户明确要求完整治理
- 任务达到 L2+ 且追踪价值明显

不要为只读分析或普通小改动强行立项。

## 用户交互约束

- 用自然语言解释正在读什么、发现了什么、准备怎么做
- 不把内部控制平面暴露成产品界面
- 不做“我先看这个，你再确认，我再看那个”的碎片化交互
- 只有在目标歧义、破坏性操作、生产影响、方向分叉明显时才暂停确认

## 编码原则

1. 先搜索，再判断。
2. 最小且可回滚的改动优先。
3. 优先复用本地已有模式。
4. 不伪造验证结果或确定性。
5. 区分事实、推断和建议。
