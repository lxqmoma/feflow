# feflow — 前端研发协作引擎

## 方法论总纲

核心目标：**做得对** — 正确、完整、可追溯地完成每一项研发活动。

六条设计原则：
1. **放得活** — 流程分级（L1/L2/L3），不一刀切
2. **管得住** — 可追踪、可核对、可恢复
3. **记得住** — 项目记忆外部化，跨会话持久
4. **接得稳** — AI 产出人类能接手
5. **啃得动** — 支持 legacy 和屎山
6. **查得到** — 证据驱动，有据可查

## 核心流程

当收到研发任务时，按以下顺序执行：

1. **检查初始化** — 确认 `.feflow/` 目录和 `project/init-config.md` 存在
2. **理解任务** — 先搜代码再下判断，识别任务类型（FEAT/MOD/BUG/HOTFIX/UI/DESIGN/CHANGE/REFACTOR/TEST/CICD/DEBT）
3. **判断级别** — L1 轻量（单文件）/ L2 标准（多文件）/ L3 重型（跨模块）/ L4 事件（线上故障）
4. **创建 Item** — 生成唯一 ID（`{TYPE}-{YYYYMMDD}-{SEQ}-{slug}`），在 `.feflow/items/` 下建工作项
5. **加载记忆** — 从 `.feflow/memory/` 读取项目配置、历史决策、团队约定
6. **扫描仓库** — 获取分支状态、技术栈、代码结构
7. **执行流程** — 按任务类型走对应 flow（需求→评审→计划→评审→实施→测试→发布）
8. **更新记忆** — 持久化关键决策、踩坑点、新发现的约束

## 记忆系统

项目记忆存储在 `.feflow/memory/` 目录下：

- `project/invariants.md` — 项目级硬性约束，任何任务都必须遵守
- `project/coding-doctrine.md` — AI 编码戒律和团队编码规范
- `modules/` — 模块级记忆（按模块名命名，记录历史问题和规避方式）
- `incidents/` — 事件记忆（线上故障、严重 bug 的教训）
- `patterns/` — 模式记忆（反模式、审查失败记录）

每次任务启动时加载相关记忆，任务结束后更新记忆，确保经验跨会话传承。

## 编码戒律（7 条）

1. **先理解再动手** — 先搜索代码、理解调用链和数据流，再开始修改
2. **最小改动** — 只做必要修改，不顺手重构无关代码，不擅自扩大范围
3. **复用优于新建** — 优先使用已有组件、工具、模式，不重复造轮子
4. **不伪造验证结果** — 不编造测试结果、运行日志或验证结论
5. **不隐藏不确定性** — 无法确认的信息标注"待验证"，不把推测伪装成事实
6. **不擅自扩大范围** — 不借题发挥重构大片无关代码，不引入未经确认的依赖
7. **保持可回滚** — 改动可追溯、可回退，commit 粒度保持最小闭环

## 门禁规则

### 编码前门禁（Gate 1）
- 项目已初始化、Item 已创建、记忆已加载、仓库已扫描
- L2+ 需求已评审且验收标准明确
- L2+ 开发计划已评审且包含历史问题对照

### 发布前门禁（Gate 2）
- 实施记录已更新、L2+ 测试报告已产出
- Commit 和分支包含 Item ID
- 代码变更与计划一致、构建通过

### 特殊门禁（Gate 3）
- Auth/Payment 改动：安全专项检查
- 公共组件改动：影响范围评估
- CI/CD 改动：回滚方案确认

## 角色体系

feflow 包含 7 个专业角色，由编排器按需调度：

| 角色 | 职责 | 调度时机 |
|------|------|----------|
| PM | 需求理解、验收标准 | L2+ 需求阶段 |
| Designer | UI/UX 设计评审 | UI/DESIGN 类任务 |
| FE | 技术方案、编码实施 | 所有任务的开发阶段 |
| Backend | API 对接、接口协议 | 前后端协同任务 |
| QA | 测试策略、回归检查 | L2+ 测试阶段 |
| Reviewer | 架构守卫、不变量检查 | L3 或代码审查阶段 |
| Researcher | 深度调研、代码考古 | 按需启用 |
