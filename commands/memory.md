# /memory

查看项目记忆概览或手动添加新的记忆条目。

## 触发方式

```
/memory              # 查看记忆概览
/memory add          # 手动添加记忆
/memory decay        # 执行记忆衰减检查与归档
```

## 执行逻辑

### 查看概览（/memory）

无参数时：

1. 读取 `.feflow/memory/` 目录下的所有记忆文件
2. 按类别分组输出概览：
   - 项目配置与约定
   - 历史决策记录
   - 团队偏好与风格
   - 已知问题与技术债
   - 复盘沉淀
3. 每条记忆展示：标题、类别、创建时间、摘要

### 手动添加（/memory add）

参数为 `add` 时：

1. 优先引导用户提供最少关键信息：
   - **类别** — 约定/决策/偏好/问题/复盘
   - **标题** — 简要描述
   - **内容** — 详细说明
   - **来源** — 可选；来自哪个 Item 或用户手动添加
2. 生成记忆文件写入 `.feflow/memory/`
3. 确认添加成功

## 前置条件

- 该能力依赖 `.feflow/` 工作区
- 若未初始化，不要只抛一句“先运行 `/init`”就结束

## 降级策略

如果 `.feflow/` 不存在：

1. 明确说明当前没有持久化记忆工作区
2. 说明这会影响什么，不会影响什么
3. 只有当用户确实想启用持久化记忆时，再建议执行 `/init`

推荐说明方式：

- “这个项目还没有启用持久化记忆，所以我现在看不到 `.feflow/memory/`。普通分析和交付不受影响；如果你想把长期约定和决策沉淀下来，再执行 `/init` 就行。”

## 说明

记忆在工作流中也会自动沉淀（如复盘产出、重要决策记录）。
`/memory add` 用于手动补充自动流程未覆盖的项目知识。

### 衰减检查（/memory decay）

按 `memory-decay` 规则检查过期记忆并按策略衰减/归档。

## 命令语义

`/memory` 是聊天命令，不是 shell 命令。

不要：

- 让用户去输入 `! /feflow:memory`
- 声称因为缺少某个 skill/tool 入口而无法查看或管理记忆

## 首轮回复契约

### `/memory`

首轮回复应直接说明会查看哪些记忆类别，并读取已有记忆。

推荐形态：

- “我先把 `.feflow/memory/` 里的项目约定、历史决策和已知问题扫一遍，再给你一个紧凑概览。”

### `/memory add`

首轮回复应只索要最少的关键信息，不要抛出一长串表单。

推荐形态：

- “我可以直接帮你记下来。给我这条记忆的标题、内容，以及它更像约定、决策还是已知问题就够了。”

### `/memory decay`

首轮回复应说明会先检查 stale 条目，再决定是否归档或降级。

不要这样开场：

- “请先运行 `/init`” 然后结束
- “我先进入记忆管理阶段”
- “请完整填写所有字段后我再开始”

暂停预算：

- 理想值：`0`
- 最大可接受：`1`，仅在用户要手动新增记忆且关键信息明显不足时
