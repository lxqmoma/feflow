# flow-legacy — 屎山/Legacy 叠加模式

**不是独立流程，而是模式叠加层。** 当 orchestrator 判断任务涉及 legacy / 高耦合区域时，在正常流程前插入额外步骤，叠加额外约束。

## 触发条件（满足任一）

- 目标模块被标记为 legacy / 高风险区
- `risk-report` 中标记的历史包袱模块
- memory 中多次出现该模块的 bug / hotfix 记录
- 代码无测试覆盖且调用链不清晰
- 用户主动声明"这块是屎山"

**产出文件路径约定**：所有产出存放在 `.feflow/items/{ITEM-ID}/` 下。

---

## 前置步骤 1：现状行为基线（叠加在原流程阶段 1 之前）

1. 页面实际行为记录：截图 / 录屏 / 关键输出快照
2. 关键状态流梳理：数据从哪来、经过什么处理、渲染到哪里
3. 依赖链和调用关系：目标文件被谁调用、调用了谁
4. Smoke check：当前功能是否正常工作
5. 记录可回滚点：commit hash / 分支 / 部署版本
6. 产出 `legacy-baseline.md`

**硬规则：基线未建立，不进入改动阶段。**

---

## 前置步骤 2：安全边界建立

1. 建立最小测试：覆盖当前行为的关键路径（不求全，求兜底）
2. 截图基线：改动前后可对比
3. 回滚点确认：确保改动可撤销
4. 记录已知 hack / workaround / TODO

---

## 前置步骤 3：影响面评估

**默认怀疑隐式依赖**，逐项检查：

| 检查维度 | 检查内容 |
|---------|---------|
| className / DOM 耦合 | 是否有外部 CSS / JS 通过 className 或 DOM 结构依赖此代码 |
| 接口字段下游依赖 | 返回字段是否被其他模块消费 |
| 公共方法复用 | 工具函数 / 公共组件是否被多处调用 |
| 全局状态 / 事件总线 | 是否通过全局 store / EventBus / window 对象通信 |
| 动态引用 / 魔法字符串 | 是否存在 `require()` / 动态 import / 字符串拼接路径 |

---

## 改动期间约束（叠加在实施阶段）

- 每次改动后运行测试，失败立即回退
- 改动范围不超出计划，发现新问题另开 Item
- 涉及 hack / workaround 的代码，改前加注释说明原因
- 不删除看似无用的代码，除非能证明无调用方
- 不顺手重构、不顺手美化、不顺手升级依赖

---

## 改动后检查（叠加在验证阶段）

1. 对比行为基线，确认功能行为未变
2. 回答**治理钩子判断题**（不是每次都重构，但每次都判断）：
   - 值得加一行注释标记技术债？
   - 值得补一个最小测试防回归？
   - 值得记一条 memory 供后续参考？
   - 值得建议后续独立治理任务？
3. 判断结果记录在 `implementation-log.md`
4. 调用 memory-update 记录改动内容、遗留风险、治理建议

---

## Legacy 硬规则汇总

1. **先观察再碰代码** — 读懂现有逻辑再动手，不基于猜测修改
2. **默认禁止无关重构** — 只做任务要求的改动
3. **先建安全边界** — 有测试或行为基线后再改，裸改就是赌博
4. **收缩影响面** — 小切口、可验证、可回滚
5. **怀疑隐式依赖** — 假设每一行代码都可能有不明显的依赖方在用
6. **每次任务结束评估** — 是否值得埋治理钩子

---

## 与其他 flow 的协作

| 宿主流程 | 叠加效果 |
|---------|---------|
| flow-feature | 强制行为基线 + 收缩改动范围 + 禁止无关重构 |
| flow-bugfix | 强制扫描隐式依赖 + 怀疑"修一个坏一片" |
| flow-refactor | 安全边界升级（覆盖率要求更高）+ 步骤粒度更细 |
| flow-cicd | 强制回滚预案验证 + 不可碰步骤标记 |

## 退出条件

模块满足以下全部条件时可申请解除 legacy 标记：
- 测试覆盖率达到项目基线
- 最近 N 个版本无 hotfix / 回归 bug
- 调用链和依赖关系已文档化
- 用户确认解除
