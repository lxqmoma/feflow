# flow-legacy — v2 Legacy 叠加模式

`flow-legacy` 不是单独流程，而是一个风险放大器。

当目标区域明显属于屎山、老系统、高耦合或低可见性区域时，在原有 Delivery / Bugfix / Refactor 流程之上叠加更保守的约束。

## 触发信号

满足任一即可考虑叠加：

- 用户明确说“这块是屎山”
- 调用链不清、测试稀缺、隐式依赖重
- 历史上多次 hotfix / 回归
- 模块存在大量 hack / workaround / FIXME
- 改动区域经常被多人高频触碰

## 叠加后的核心要求

1. **先观察再碰代码**
2. **先建最小安全边界**
3. **默认怀疑存在隐式依赖**
4. **改动面尽量收缩**
5. **不做顺手重构**

## 最小动作

至少补上这些动作：

- 现状行为快照
- 关键调用链梳理
- 最小 smoke check
- 回滚点确认

这些信息可以：

- 在有 `.feflow/` 时沉淀为记录
- 在无 `.feflow/` 时以内联摘要存在

## Legacy 区的额外验证

- 每次关键修改后尽快验证
- 重点看旧行为有没有被误伤
- 无法证明无依赖时，不要贸然删代码

## 何时可以解除 legacy 叠加

- 已有最小测试安全网
- 调用链和约束基本清晰
- 最近没有持续热修或回归
- 当前任务已不再依赖“猜系统”
