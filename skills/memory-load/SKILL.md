# memory-load: 任务前记忆加载

## 触发时机

由 orchestrator 在创建 Item 后、分配执行 agent 前调用。
目的是将项目积累的历史经验注入当前任务上下文，避免重复踩坑。

## 输入

orchestrator 传入当前 Item 的基本信息：

- `item_id` — 工作项标识（如 `FEAT-0012`）
- `item_type` — 类型（feat / fix / refactor / hotfix / chore）
- `modules` — 涉及模块列表（如 `["auth", "payment"]`）
- `risk_level` — 风险级别（low / medium / high / critical）
- `title` — 任务标题
- `description` — 任务描述（可选）

## 加载步骤

### 第一步：读取全局不变量

读取 `.feflow/memory/project/invariants.md`。
该文件记录项目级的硬性约束，任何任务都必须遵守，不可跳过。

典型内容包括：
- 禁止直接操作的文件或目录
- 必须保留的接口契约
- 不可变更的数据结构
- 构建和部署的强制要求

若文件不存在，跳过此步，不报错。

### 第二步：读取 AI 编码戒律

读取 `.feflow/memory/project/coding-doctrine.md`。
该文件记录团队沉淀的编码规范和 AI 协作约定，适用于所有开发类任务。

典型内容包括：
- 命名约定和代码风格要求
- 禁止使用的模式或反模式
- AI 生成代码的审查要求
- 测试覆盖的底线标准

若文件不存在，跳过此步，不报错。

### 第三步：按模块查找模块记忆

遍历 `.feflow/memory/modules/` 目录，查找与当前任务涉及模块匹配的记忆文件。

匹配规则：
1. 文件名匹配 — 文件名包含模块名（如 `auth.md` 匹配模块 `auth`）
2. frontmatter scope 匹配 — 文件 frontmatter 中 `scope` 字段包含目标模块名
3. 仅加载 `status: active` 的记忆条目

对每个匹配的文件，提取其中的记忆条目。每条记忆包含：
- 场景描述
- 曾经遇到的问题
- 根因分析
- 规避方式

### 第四步：查找相关事件记忆

扫描以下目录，查找与当前任务相关的事件记录：

1. `.feflow/memory/incidents/` — 线上事故和严重问题的复盘记录
2. `.feflow/memory/patterns/` — 反复出现的问题模式

匹配条件（满足任一即命中）：
- 事件的 `scope` 包含当前任务涉及的模块
- 事件的 `type` 与当前任务类型相同
- 事件内容中出现当前任务涉及的模块名关键词

对于 `risk_level` 为 high 或 critical 的任务，放宽匹配条件，额外加载：
- 所有 `type: incident` 的记忆（无论模块是否匹配）
- 所有 `scope: global` 的记忆

### 第五步：查找历史相似 Item

扫描 `.feflow/items/` 目录（若存在），查找与当前 Item 相似的历史任务。

相似判定条件（满足两项及以上即视为相似）：
- 涉及相同模块
- 任务类型相同
- 标题或描述中包含相同关键词

从相似 Item 的完成记录中提取：
- 实际耗时与预估偏差
- 遇到的未预见问题
- 最终采用的方案（若与初始方案不同）

## 筛选条件

所有记忆条目在加载后需经过筛选，仅保留有效记忆：

1. **状态过滤** — 仅保留 `status: active` 的记忆，跳过 `status: archived` 或 `status: superseded`
2. **scope 匹配** — 记忆的 scope 必须与当前任务的模块列表有交集，或 scope 为 `global`
3. **内容相关性** — 记忆内容中需包含当前任务涉及的模块名、关键技术栈名称或相关业务术语
4. **去重** — 同一 `memory_id` 仅保留一次，避免跨目录重复加载

## 产出格式

加载完成后，生成 `00-memory-brief.md` 文件，写入当前 Item 的工作目录。

完整格式如下：

```markdown
---
item_id: FEAT-0012
generated_at: 2026-04-08T10:30:00+08:00
memory_count: 5
risk_level: high
modules:
  - auth
  - payment
---

# 记忆摘要

## 全局不变量

> 以下为项目级硬性约束，本次任务必须遵守。

- **[INV-001]** 用户表主键不可变更，所有关联查询依赖 `user_id` 字段
- **[INV-003]** 支付回调接口签名验证逻辑禁止修改，变更需走安全评审
- **[INV-005]** 构建产物必须通过 CI 流水线生成，禁止本地构建后直接部署

## 编码戒律

> 以下为团队编码规范和 AI 协作约定。

- **[DOC-002]** 所有对外接口必须保持向后兼容，新增字段使用可选类型
- **[DOC-004]** 数据库查询禁止在循环中执行，必须批量操作
- **[DOC-007]** AI 生成的代码必须经过人工审查，不可直接合并到主分支

## 模块记忆

### auth 模块

- **[MEM-A001]** 登录态刷新存在竞态条件
  - 场景：多个请求并发触发 token 刷新
  - 问题：刷新接口被重复调用，旧 token 被提前作废导致部分请求 401
  - 根因：刷新逻辑未加锁，多个请求各自独立发起刷新
  - 规避方式：使用请求队列，首个请求负责刷新，其余请求等待刷新结果

### payment 模块

- **[MEM-P003]** 支付状态轮询超时处理不当
  - 场景：用户完成支付后，前端轮询支付结果
  - 问题：网络抖动导致轮询超时，前端显示支付失败但实际已成功
  - 根因：超时后直接判定为失败，未做二次确认
  - 规避方式：超时后调用订单查询接口做最终确认，不直接判定失败

## 历史事件

- **[INC-007]** 2026-02-15 支付回调签名验证绕过事故
  - 影响范围：payment 模块
  - 根因：依赖库升级后签名算法默认值变更，未被测试覆盖
  - 教训：依赖升级后必须运行支付相关的完整回归测试
  - 当前状态：已修复，已补充测试用例

## 本次必须额外检查

> 根据本次任务的模块和风险级别，以下事项需额外关注：

1. auth 模块的 token 刷新逻辑是否仍存在竞态风险
2. payment 模块的任何接口变更是否影响回调签名验证
3. 涉及 high 风险任务，建议完成后触发完整回归测试
```

## 记忆库为空时的输出

当 `.feflow/memory/` 目录不存在或目录下无任何记忆文件时，生成以下最小化的 `00-memory-brief.md`：

```markdown
---
item_id: FEAT-0012
generated_at: 2026-04-08T10:30:00+08:00
memory_count: 0
risk_level: medium
modules:
  - auth
---

# 记忆摘要

> 项目刚初始化，暂无历史记忆记录。

## 全局不变量

暂无记录。建议在任务完成后，通过 memory-update 沉淀本次经验。

## 编码戒律

暂无记录。建议团队逐步补充编码规范到 `.feflow/memory/project/coding-doctrine.md`。

## 模块记忆

暂无记录。本次任务涉及的模块尚无历史经验沉淀。

## 历史事件

暂无记录。

## 本次必须额外检查

暂无基于历史经验的检查项。建议按常规开发流程执行。
```

## 执行约束

1. memory-load 是只读操作，不修改任何记忆文件
2. 加载过程中遇到文件解析错误（如 frontmatter 格式不合法），跳过该文件并在产出中标注警告
3. 单次加载的记忆条目总数上限为 30 条，超出时按以下优先级截断：
   - 全局不变量（始终保留）
   - 编码戒律（始终保留）
   - 事件记忆（按时间倒序，优先保留近期）
   - 模块记忆（按匹配度排序，优先保留强匹配）
   - 历史相似 Item（按相似度排序，最多保留 5 条）
4. 产出文件 `00-memory-brief.md` 的文件名前缀 `00-` 确保它在 Item 工作目录中排在最前，便于执行 agent 优先读取
5. 加载耗时不应超过 5 秒，若记忆库文件过多，优先加载 scope 精确匹配的文件

## 与其他 skill 的关系

- **上游**：orchestrator 创建 Item 后调用本 skill
- **下游**：执行 agent 读取 `00-memory-brief.md` 开始工作
- **对称**：memory-update 在任务完成后写入新记忆，供后续任务的 memory-load 检索
