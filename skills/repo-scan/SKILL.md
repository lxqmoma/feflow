# repo-scan — v2 仓库画像与风险扫描

`repo-scan` 的目标是快速建立事实级仓库画像，而不是在每次任务前机械跑一次 Git 体检。

## 适用场景

- 用户显式执行 `/feflow:scan`
- 需要快速理解仓库结构、技术栈和高风险区域
- Delivery 或 Incident 任务开始前，需要更广的仓库画像

不建议：

- 对所有任务无差别自动执行完整扫描
- 把扫描结果当成强制前置文档

## v2 设计原则

1. 即使没有 `.feflow/`，也应该能工作。
2. 即使不是 Git 仓库，也至少应该能完成项目结构扫描。
3. 输出应以用户可行动的信息为中心，而不是完整机械表格堆砌。
4. 只有在工作区存在时，才把结果持久化。

## 扫描层次

### 第一层：项目结构与技术栈

始终可执行，重点包括：

- 关键配置文件
- 目录拓扑
- 主要框架、语言、构建工具
- 测试、lint、CI 存在情况
- 明显的高风险目录或核心入口

### 第二层：Git 状态

仅当当前目录是 Git 仓库时执行，重点包括：

- 当前分支
- 默认分支
- 工作区脏状态
- ahead / behind
- 最近提交和高频改动区域

### 第三层：风险热点

结合目录结构和最近改动识别：

- 高频修改文件
- 高频修改目录
- 多作者冲突热点
- 发布敏感区域

### 第四层：治理状态

仅当 `.feflow/` 存在时执行：

- 已跟踪工作项概况
- 状态漂移
- 未登记改动
- 可补录项

## 输出内容

输出应至少回答这些问题：

1. 这是个什么类型的项目？
2. 项目主入口和主目录在哪？
3. 哪些区域值得优先关注？
4. 当前 Git 状态是否健康？
5. 是否存在明显治理风险或历史包袱？

## 持久化策略

### `.feflow/` 存在

可将结果写入 `00-repo-scan.md`，供后续治理能力使用。

### `.feflow/` 不存在

直接输出内联摘要，不把初始化作为阻塞。

## 告警策略

只输出有行动意义的告警，例如：

- 当前分支严重落后默认分支
- 工作区很脏，可能污染判断
- 最近热点区域存在多人高频修改
- 发布敏感区域近期有剧烈变更
- 已跟踪工作项与真实提交状态明显漂移

不要为每一项轻微异常都制造流程负担。

## 错误处理

### 非 Git 项目

继续做项目结构扫描，跳过 Git 相关部分。

### `.feflow/` 不存在

继续扫描，不报硬错误。

### 配置或 Item 文件损坏

记录警告并跳过，不中止主扫描。

## 性能要求

1. 默认先做轻量扫描
2. 大仓库优先输出高信号摘要，而不是完整明细
3. 只有在用户明确需要时，再扩展到更深层扫描
