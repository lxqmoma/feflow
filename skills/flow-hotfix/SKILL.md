# flow-hotfix — v2 紧急恢复流程

`flow-hotfix` 对应 Incident 模式。

它不是常规 bugfix 的加速版，而是面向真实事故的恢复路径。

核心目标按优先级排序：

1. 恢复业务
2. 控制影响面
3. 保留回滚能力
4. 在恢复后补足根因和治理材料

## 何时进入 hotfix

满足任一即可：

- 核心流程不可用
- 大量用户受影响
- 发布后出现严重回归
- 存在安全、权限、数据一致性等紧急风险
- 时间敏感，不能等待常规修复节奏

若只是普通 bug，回到 `flow-bugfix`。

## 硬规则

1. 不要求在恢复前创建完整文档。
2. 不要求在恢复前完成完整根因分析。
3. 不要求 `.feflow/` 初始化后才能开始。
4. 允许优先选择回滚、降级、开关关闭，而不是执着于 fix-forward。
5. 禁止借 hotfix 顺手重构、换方案、引入新依赖或扩大范围。

## 恢复决策顺序

优先按以下顺序选择恢复手段：

1. **回滚**：如果最近变更明确且回滚成本最低
2. **降级/开关**：如果关闭部分能力能快速恢复核心路径
3. **最小修复**：如果可在可控范围内快速 fix-forward

不要默认“写代码修复”就是第一选择。

## 执行节奏

### 第一步：确认影响和优先级

快速确认：

- 影响功能
- 影响用户范围
- 是否有替代路径
- 是否需要立即停止发布或合并

### 第二步：快速定位

允许使用：

- 最近改动
- 发布差异
- 日志/报错
- 高概率链路代码

这一步的目标是“找到可信恢复方向”，不是产出完美根因报告。

### 第三步：执行最小恢复动作

恢复动作可以是：

- revert / rollback
- feature flag 关闭
- fallback 逻辑恢复
- 配置修正
- 最小代码补丁

### 第四步：最小验证

至少验证：

1. 原事故现象是否缓解
2. 核心路径是否恢复
3. 新动作是否引入更大故障

可选验证：

- lint / type-check
- 核心冒烟测试
- 最小自动化测试

如果时间成本过高，可以跳过非关键验证，但必须在输出中明确说明。

### 第五步：恢复后补录

恢复完成后，再补：

- 根因分析
- 残余风险
- 后续长期修复项
- 记忆更新和证据沉淀

若 `.feflow/` 工作区存在，应把这些信息沉淀进去。
若不存在，应至少以内联形式给出补做清单。

## 用户交互规则

对用户输出只聚焦：

- 当前影响
- 你准备采取的恢复动作
- 为什么这是最快且最安全的路径
- 是否需要立即确认某个高风险选择

不要在事故处理中输出冗长流程播报。

## 验证强度

| 场景 | 最低要求 |
|------|----------|
| 回滚 | 确认目标版本和恢复结果 |
| 降级 | 确认核心路径恢复，次要能力降级可接受 |
| 最小补丁 | 确认原问题消失，关键路径可用 |

## 恢复后的治理要求

只有恢复完成后，才建议补以下材料：

- incident note
- root cause
- rollback note
- follow-up tasks
- project memory update

这些材料很重要，但不应阻塞恢复动作本身。

## 异常处理

- 如果 fix-forward 无效，优先回滚
- 如果无法快速定位，优先降级
- 如果没有安全恢复路径，及时向用户暴露风险并请求关键决策
