# flow-bugfix — v2 缺陷修复流程

`flow-bugfix` 用于非事故型 bug 修复。

核心原则：

1. 先区分现象、触发条件和根因
2. 不做只治表象的补丁式修复
3. 修复节奏按风险控制，不按模板广播

## 何时使用

适用于：

- 可在正常研发节奏中处理的 bug
- 不要求立刻恢复线上服务的缺陷
- 需要根因分析和回归验证的修复

不适用于：

- 线上事故
- 时间敏感的热修复
- 需要优先止血的发布故障

这类情况应转入 `flow-hotfix`。

## 硬规则

1. 不要把直接报错位置当成根因。
2. 不要默认把所有 bug 都拆成“分析文档 + 方案确认 + 修复 + 回归”四次对话。
3. L1 bug 允许直接修复并验证。
4. 只有在存在明显方案分叉或高风险影响面时，才需要单独确认修复策略。
5. 如果 `.feflow/` 不存在，流程仍可执行，只是无法持久化治理材料。

## 通用执行顺序

1. 收集现象和复现条件
2. 读取相关代码，定位调用链与数据流
3. 区分表象与根因
4. 判断修复级别
5. 实施修复
6. 验证修复点和相邻回归面
7. 输出根因、改动和残余风险

## L1：直接修复

适用于：

- 局部缺陷
- 单文件或单链路修复
- 根因较清晰

执行方式：

1. 最小复现或静态定位
2. 形成根因假设
3. 直接修复
4. 运行最小可信验证
5. 汇报修复结论

默认不要求：

- 单独的 bug 分析文档
- 单独的方案选择环节
- Item

## L2：结构化修复

适用于：

- 涉及多文件
- 影响一个以上模块或状态面
- 回归风险明显高于局部修复

执行方式：

1. 输出简短的诊断摘要：
   - 现象
   - 根因
   - 影响面
   - 修复方向
2. 若工作区存在，可加载相关记忆
3. 只有在以下场景才建议创建 Item：
   - 修复会跨多轮会话
   - 这是高复发问题，需要沉淀证据
   - 需要对账或追踪后续修复项
4. 实施修复
5. 做针对修复点和相邻链路的回归验证

## L3：高风险修复

适用于：

- 根因位于共享基础设施、全局状态、路由、权限、构建或公共接口
- 修复可能引入新的系统性风险
- 需要跨模块协调

执行要求：

1. 必须给出明确根因摘要
2. 必须说明影响面
3. 必须给出最小可行回滚意识
4. 若 `.feflow/` 可用，应创建 Item 并沉淀关键记录
5. 验证必须覆盖修复点、相邻模块和高风险边界

## 修复策略选择

只有在确实存在方向分叉时，才向用户展示多个方案。

适合停下来确认的情况：

- 最小修复和稳妥修复之间成本差异明显
- 修复方案会改变既有行为或边界
- 修复涉及共享 API 或公共组件契约

不适合停下来确认的情况：

- 明显的局部 bug，修复方向已经清楚
- 用户目标就是“把问题修好”
- 继续阅读代码即可消除不确定性

## 验证要求

### 所有级别都至少验证

1. 原始现象是否消失
2. 修复点是否符合预期
3. 是否引入了明显新错误

### L2/L3 额外验证

- 相邻链路
- 历史高频回归点
- 异常输入、边界状态、空值或并发状态

### 证据强度

| 级别 | 推荐证据 |
|------|----------|
| L1 | 复现消失 + 最小验证 |
| L2 | 修复验证 + 自动化检查摘要 |
| L3 | 修复验证 + 回归验证 + 关键风险说明 |

## 推荐产物

只有在治理有收益时才写文件。

| 场景 | 推荐产物 |
|------|----------|
| L1 | 内联诊断与验证摘要 |
| L2（有工作区） | bug 诊断摘要、实施记录、测试摘要 |
| L3 | 根因记录、测试证据、必要时复盘或 follow-up |

## 收尾

缺陷修复完成后，重点沉淀这些信息：

- 根因
- 为什么之前没发现
- 哪些边界容易再次出问题
- 是否需要后续长期修复项

不要为了“流程闭环”补一堆低价值文档。

## 异常分流

- 无法复现但影响真实存在：记录已知条件，继续静态分析，不必立刻停机等待
- 根因疑似第三方依赖：说明证据、给出 workaround，再决定是否继续
- 如果问题已升级为线上恢复场景：转 `flow-hotfix`
