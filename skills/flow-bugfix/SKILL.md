# flow-bugfix — Bug 修复流程

当 orchestrator 将任务分类为 BUG 时，按本流程推进。

核心原则：**先定位根因，再动手修复**。不做表面补丁，不顺手扩大改动范围。

## 前置条件

- feflow 已初始化（`.feflow/` 目录存在且 `project/init-config.md` 配置完整）
- orchestrator 已完成任务分类并生成 Item（包含唯一 ID，如 `BUG-001`）
- memory-load 已执行，当前会话拥有项目记忆上下文

## 分支命名

```
bugfix/{ITEM-ID}-{slug}
```

示例：`bugfix/BUG-001-login-redirect-loop`

---

## 阶段 1：现象收集

从用户输入提取：错误现象、复现步骤、影响范围、环境信息。查询 memory 中相似/相关历史 bug。信息不足时追问，但能合理推断的先推断并标注假设。产出结构化描述写入 Item 上下文。

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 收集深度 | 基本现象描述即可 | 需要复现步骤和环境信息 | 完整收集 + 历史关联分析 |
| 历史查询 | 快速扫描 | 必须查询 | 深度查询 + 相似问题对比 |
| 追问策略 | 信息足够则不追问 | 关键信息缺失时追问 | 主动确认边界场景 |

---

## 阶段 2：根因分析

定位根本原因，而非仅找到出错位置。推断出错模块 → 追踪调用链和数据流 → 区分**根因**（源头条件）和**表象**（直接报错行） → 验证根因假设。

**bug-analysis.md 结构**（产出至 `.feflow/items/{ITEM-ID}/bug-analysis.md`）：

```markdown
# Bug 分析 — {ITEM-ID}
## 现象摘要
## 复现条件（操作步骤 / 触发条件 / 环境）
## 根因定位（根因描述 / 涉及文件 / 代码路径）
## 表象 vs 根因
## 影响范围（受影响功能 / 受影响场景）
## 历史关联（相关历史 bug / 是否为回归）
```

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 分析深度 | 定位到出错代码即可 | 追溯调用链找到根因 | 根因 + 系统性分析 |
| 分析报告 | 简要记录 | 完整 bug-analysis.md | 完整报告 + 架构层面反思 |
| 历史关联 | 可省略 | 必须查询 | 深度关联 + 是否为系统性问题 |

---

## 阶段 3：修复方案

**目标**：制定修复方案，在"最小修复"和"稳妥修复"之间做出选择。

**执行逻辑**：

1. 基于根因分析，制定两种方案（L1 可直接给最优方案）：
   - **方案 A — 最小修复**：只改出错点，改动小风险低，可能治标不治本
   - **方案 B — 稳妥修复**：加固相关逻辑，从根本解决，改动较大需更多测试
2. 评估优劣（改动范围、是否引入新问题、是否根本解决），给出推荐和理由
3. L2+ 呈现给用户选择

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 方案数量 | 直接给出最优方案 | 提供两种方案 | 提供两种方案 + 长期改进建议 |
| 用户确认 | 必须确认（一句话确认即可） | 必须确认 | 必须确认 |
| 方案深度 | 直接说改哪里 | 改动 + 影响分析 | 改动 + 影响 + 系统性防范 |

---

## 阶段 4：修复实施

创建分支 `bugfix/{ITEM-ID}-{slug}` → 调用 quality-gate 门禁检查 → 按方案修复（只改该改的，不顺手重构，不引入计划外依赖） → 补充测试用例（superpowers 可用时建议 TDD） → 记录至 `implementation-log.md`。

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 门禁检查 | 基础检查 | 完整检查 | 完整检查 |
| 补充测试 | 可选 | 建议补充 | 必须补充 |
| 实施日志 | 简要 | 按步骤记录 | 按步骤记录 + 决策依据 |
| 代码审查 | 可省略 | 建议 code-review | 必须 code-review |

---

## 阶段 5：回归测试

**目标**：确认 bug 已修复且未引入新问题。

**执行逻辑**：

1. 按复现步骤验证 bug 已修复
2. 运行自动化测试：lint / type-check / unit test / e2e
3. **回归检查**：修改文件涉及的其他功能、memory 中历史 bug、边界场景（空值/异常输入/并发）
4. 产出至 `.feflow/items/{ITEM-ID}/role-qa/test-report.md`（含修复验证、自动化测试、回归检查、结论）

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 验证范围 | 修复点 + lint | 修复点 + 全套自动化测试 | 全套 + 相邻模块 + 边界场景 |
| 回归检查 | 快速扫描 | 必须检查 | 深度检查 + 相邻模块 |
| 报告详细度 | 简要 | 完整 | 完整 + 根因验证说明 |

---

## 阶段 6：收尾

**目标**：合并代码，沉淀经验。

**执行逻辑**：

1. 创建 PR：标题 `fix({scope}): {描述} [{ITEM-ID}]`，描述含 bug 现象/根因/修复方案/测试结果
2. 调用 memory-update 写入记忆：根因和修复方式、涉及模块注意事项、值得警惕的模式
3. 更新 Item 状态为 `completed`
4. **线上问题额外复盘**（生产环境 bug 必须执行）：产出 `.feflow/items/{ITEM-ID}/postmortem.md`（时间线/影响范围/根因/防范措施），复盘结论标记为高优先级记忆

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| PR 创建 | 可直接合并 | 必须创建 PR | 必须创建 PR |
| 记忆更新 | 简要记录 | 完整记录 | 完整记录 + 模式归纳 |
| 复盘 | 仅线上问题 | 仅线上问题 | 线上必须 + 复杂 bug 建议 |

---

## 状态流转

```
created
  → bug_analyzing（阶段2：根因分析中）
  → plan_drafted（阶段3：方案已起草）
  → plan_approved（阶段3：方案已确认）
  → implementing（阶段4：修复实施中）
  → testing（阶段5：回归测试中）
  → completed（阶段6：收尾完成）
```

## 异常处理

- **无法复现**：记录尝试的复现步骤和环境，询问用户补充信息。若确认无法复现，标记为 `cannot_reproduce` 并关闭
- **根因在第三方依赖**：记录分析结论，评估 workaround 方案，必要时提 issue 到上游
- **修复引入新问题**：回退修复，回到阶段 2 重新分析，记录失败原因
- **线上紧急 bug**：如需立即止血，转入 flow-hotfix 流程
