# flow-test-task — v2 测试任务流程

`flow-test-task` 处理测试建设类任务，而不是业务实现任务。

核心原则：

1. 明确测试目标
2. 优先遵循项目现有测试栈和风格
3. 测试任务里发现的业务问题应分流，不要顺手改业务代码

## 适用场景

- 给老模块补测试
- 建立 e2e 或集成测试
- 做专项测试验收
- 为高风险模块补安全网

## 硬规则

1. 默认只做测试相关改动。
2. 发现业务 bug 时记录并分流。
3. 覆盖率不是唯一目标，风险覆盖更重要。

## 执行重点

### 范围定义

先明确：

- 目标模块
- 测试类型
- 风险优先级
- 明确不覆盖什么

### 用例设计

优先覆盖：

- 正常路径
- 高风险边界
- 异常输入
- 异步或并发行为

### 实施

- 遵循现有测试工具链
- 避免为了让测试通过而篡改业务逻辑

## 输出重点

完成后至少说明：

- 新增了什么测试
- 覆盖了哪些风险
- 还遗留哪些空白
- 过程中发现了哪些非测试问题

## 何时需要确认

- 测试目标明显扩张
- 模块过于不可测，需要先重构
- 用户要求测试任务顺带修 bug
