# backfill — v2 事后补录

`backfill` 用于把已经发生的工作反向沉淀成治理记录。

它的目标是提高可追踪性，而不是在用户已经把事做完后再强行补一整套流程仪式。

## 适用场景

- 代码已改完，但没有对应 Item
- PR 已经合并，希望补回治理记录
- 想把历史改动纳入记忆和证据体系

## 前置条件

- 当前目录是 Git 仓库
- `.feflow/` 工作区存在

如果没有 `.feflow/`：

- 不要报硬错误追着用户初始化
- 可以先给出“可补录范围”摘要
- 用户确实需要治理记录时，再建议初始化后补录

## 最小补录原则

补录时优先恢复这些关键信息：

- 这次改动是什么
- 为什么改
- 改了哪些文件/模块
- 有哪些可用证据
- 仍缺少哪些治理材料

不要为了补录而假装存在当时并没有发生的评审和计划过程。

## 产出策略

应优先生成：

- 简要需求/背景摘要
- 实际改动清单
- 可获得的测试或 CI 证据
- 缺失项清单

缺失的就明确标注缺失，不要伪造完整闭环。

## 用户交互

用户通常最关心的是：

- 哪些历史改动值得补录
- 补录后能得到什么价值
- 哪些信息已经无法恢复

不要把 backfill 变成“未登记改动审判流程”。
