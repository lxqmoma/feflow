# project-init：v2 治理工作区初始化

## 触发条件

- 用户执行 `/init` 或明确要求初始化 feflow
- 用户明确需要持久化 Item / Memory / Evidence
- 检测到 `.feflow/` 存在但缺少基础配置，需要补全

## 前置检查

1. 当前目录应是有效项目目录；Git 和 `package.json` 是加分项，不是绝对门槛
2. 若 `.feflow/` 已存在，优先补全，不默认覆盖
3. 初始化目标是建立治理工作区，而不是强行完成一次全量项目研究

## 执行姿态

`/init` 本身就是用户的授权信号，不应再被拆成“是否允许我初始化”的二次确认。

现实约束：

- 某些宿主里的 slash command 很容易在第一条助手消息后直接结束当前回合
- 因此初始化不能依赖“我先解释，再下一条真正执行”的节奏
- 对 `/init`，应把工具调用视为比铺陈更高优先级的动作

规则：

1. 对于有界、非破坏性的最小初始化，默认连续执行
2. 不把内部 `project-init` 名称暴露成外部运行时依赖
3. 不因为缺少某个 skill/router 入口就宣告初始化无法进行
4. 如果宿主支持文件写入，直接落盘；如果宿主不支持，明确说明这是宿主限制，并给出可直接应用的文件结果
5. 初始化后的补充校准永远不阻塞最小工作区落地
6. 如果宿主暴露了正常文件工具，则必须在当前这一回合完成真实创建/补全；不能只做口头承诺后直接结束
7. 不要求用户再次发送 `/init` 或 `/feflow:init` 来触发真正执行
8. 在 Claude Code 类宿主中，优先使用 `Bash` 创建目录、`Write/Edit` 生成基础文件、`Read/Bash` 校验结果
9. 如果命令层已经通过 shell dispatch 完成了最小创建/补全，则把它当成已执行事实：直接基于结果汇报，不再用“我先初始化 …”做开场

## 执行流程

### 阶段一：创建最小工作区

```bash
mkdir -p .feflow/{project,items,memory/{project,modules,incidents,patterns},templates}
```

目录用途：

- `project/` — 项目级配置与画像
- `items/` — 被追踪任务
- `memory/` — 持久化约束与经验
- `templates/` — 模板副本或扩展

执行要求：

- 若目录缺失，直接创建
- 若目录已存在，继续检查缺失文件
- 不把“我将创建这些目录”当成完成信号；只有真实文件系统状态发生变化或被验证为已存在，才算这一阶段完成
- 如果宿主允许 Bash，优先一次性完成目录骨架创建，而不是把目录列表重新说给用户

### 阶段二：生成最小配置

初始化时只要求产出足够支持治理的配置，不要求一次把所有信息问全。

`init-config.md` 至少包含：

- 工作区形态
- 风险治理偏好
- 高风险区域（如能识别）
- AI 禁区（如能识别）
- 初始化时间

若某些值无法确认：

- 可使用保守默认值
- 可标记为 `unknown`
- 不把初始化流程卡死

### 阶段三：生成初始项目画像

可基于轻量扫描生成 1-3 个高价值文件，例如：

- `project-profile.md`
- `architecture-report.md`
- `risk-report.md`

重点是建立“以后能持续更新的基线”，不是一次性做满全量审计。

### 阶段四：初始化基础记忆

在 `.feflow/memory/project/` 下创建基础文件：

- **invariants.md** — 项目不变量（空模板），含架构不变量、数据不变量、流程不变量三个分区
- **coding-doctrine.md** — 项目级协作和编码戒律

### 阶段五：可选补充校准

如果关键治理信息缺失，可在初始化**完成后**补充极少量问题。

优先只问这类高价值问题：

1. 哪些区域是高风险区
2. AI 不能碰哪些内容
3. 上线前需要什么级别的确认

不要把初始化变成冗长问卷。
更不要把这些问题当成最小初始化的前置门槛。

## 输出总结

初始化完成后输出：

- 工作区是否已可用
- 创建了哪些目录和基础文件
- 当前仍未知但以后可补充的配置
- 如何继续使用 `assist / task / incident / scan`

禁止把以下内容当成完成：

- 只复述即将执行的步骤
- 只解释为什么这是一个初始化命令
- 要求用户再发一次同样的命令
- 把第一条开场白当成“这一轮已经完成初始化”
- 在 dispatch 已经落盘后，仍然用未来时态假装工作还没开始

## 错误处理

| 场景 | 处理 |
|------|------|
| `package.json` 不存在 | 继续，以通用项目方式初始化 |
| 不在 Git 仓库中 | 继续初始化，标记 Git 能力不可用 |
| `.feflow/` 已存在且完整 | 以补全/修复模式运行 |
| 扫描失败 | 继续建立最小工作区 |
| 用户未补充校准信息 | 用保守默认值初始化 |
| 现有 `.feflow/` 文件会被模板覆盖 | 暂停一次，只对覆盖/合并策略做最小确认 |
| 首轮回复后还没落盘任何内容 | 不要结束当前回合，继续执行最小创建/修复 |
