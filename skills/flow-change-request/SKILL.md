# flow-change-request — v2 需求变更流程

`flow-change-request` 处理的是“已有任务中途发生变化”。

核心原则：

1. 原始需求不应被悄悄覆盖
2. 变更需要说明影响
3. 只更新真正被影响的执行面

## 适用场景

- 用户中途改口
- 需求范围新增或删除
- 技术发现迫使方案调整
- 外部依赖变化导致既有计划失效

## 与 v1 的区别

v2 不要求把每次变更都重跑完整需求流。

默认做法是：

1. 读原目标
2. 描述变更点
3. 评估影响
4. 更新当前执行策略
5. 只在真正高风险时停下来确认

## 硬规则

1. 不要悄悄覆盖原目标。
2. 不要为了记录变更而强迫重建整套文档。
3. 变更如果会推翻已完成工作，必须明确说出来。
4. 多次变更时，应保留版本意识或至少保留变更脉络。

## 最小变更记录

至少说明四件事：

- 原目标是什么
- 现在变了什么
- 影响了哪些步骤或文件
- 是否需要返工或回滚部分已完成工作

## 风险分级

| 级别 | 特征 | 默认策略 |
|------|------|----------|
| L1 | 小范围调整，不影响主体策略 | 直接更新执行路径 |
| L2 | 影响多个步骤或部分已完成实现 | 更新计划并说明返工面 |
| L3 | 改变范围、架构、发布时间或共享边界 | 强治理并确认关键决策 |

## 何时需要确认

- 变更会推翻已完成工作
- 变更会扩大范围或延期
- 变更会改变用户体验、接口契约或架构边界

其他情况下，可以直接更新计划并继续执行。
