# flow-change-request — 需求变更流程

当 orchestrator 将任务分类为 CHANGE 时，按本流程推进。

核心原则：**变更不是重写需求，而是在原需求基础上叠加修改。原需求文档不可覆盖，计划有版本可追溯。**

## 前置条件

- feflow 已初始化，memory-load 已执行
- 必须存在已有 Item（已进入阶段 2+），原需求 `role-pm/requirement-brief.md` 已存在
- 沿用原 Item 及其分支，不新建 Item

**产出文件路径约定**：所有产出存放在 `.feflow/items/{ITEM-ID}/` 下。

---

## 阶段 1：读取原需求

1. 找到关联的原始 Item，确认当前状态和已有产出
2. 读取 `role-pm/requirement-brief.md`，理解原始需求范围和验收标准
3. 确认变更来源：用户反馈 / PM 决策 / 技术发现 / 外部依赖变化

---

## 阶段 2：生成变更单

产出 `change-request.md`，内容必须覆盖：

| 维度 | 必填内容 |
|------|---------|
| 变更来源与原因 | 谁提出、为什么变 |
| 变更内容 | 逐条列出：新增 / 修改 / 删除 |
| 对需求的影响 | 哪些验收标准变了 |
| 对计划的影响 | 工时、步骤调整 |
| 对测试的影响 | 用例新增 / 修改 / 废弃 |
| 对发布时间的影响 | 是否延期 |
| 对已完成代码的影响 | 需改 / 废弃 / 回退 |
| 是否需要返工 | 已实施部分是否要推翻重做 |
| 变更风险 | 引入的新风险点 |

**硬规则：禁止直接修改或覆盖 `role-pm/requirement-brief.md`。原需求只读。**

---

## 阶段 3：变更评审

**必须经用户确认。AI 不可自行跳过。**

呈现变更单，确认：内容准确性、影响完整性、成本可接受性。

- **批准** → 进入阶段 4
- **修改** → 回到阶段 2 修订
- **拒绝** → 记录原因，关闭变更流程

---

## 阶段 4：更新开发计划

**保留版本号，不覆盖原版本**：

- `dev-plan.md` → 重命名为 `dev-plan-v1.md`（只读归档）
- 新建 `dev-plan-v2.md` 作为当前版本
- 多次变更依次递增：v3、v4...

新版计划必须：
- 标注因变更而新增 / 修改 / 删除的步骤
- 标注需返工的已完成步骤
- 参照 flow-feature 阶段 3 的编写规范

---

## 阶段 5：计划评审

**未确认不可实施。** 呈现新版计划，用户确认后进入实施。

- **通过** → 进入阶段 6
- **修改** → 回到阶段 4
- **退回** → 回到阶段 2

---

## 阶段 6：实施

参照 flow-feature 阶段 5。若 Item 已有实施产出，优先评估已完成代码的影响：
- 受影响的文件和逻辑
- 是否需回退提交
- 已写测试是否需修改或废弃

---

## 阶段 7：测试验证

参照 flow-feature 阶段 6。额外关注变更引入的回归风险。

---

## 阶段 8：收尾

调用 memory-update 写入：变更原因、影响范围、返工内容、踩坑点。更新 Item 状态为 `completed`。

## 状态流转

```
change_requested → change_approved → plan_updated → plan_approved → implementing → testing → completed
```

若 Item 已进入实施阶段，`plan_updated` 前增加 `impact_assessed`。

## 异常处理

- **变更与原需求矛盾**：标注矛盾点，用户裁决以哪个为准
- **变更导致架构调整**：升级为 L3，调用 architect agent
- **多次变更累积**：每次独立生成变更单，版本号持续递增，不合并变更单
- **用户未响应评审**：提醒一次后等待，不自行推进
