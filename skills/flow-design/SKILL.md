# flow-design — v2 设计落地流程

`flow-design` 处理两类任务：

1. 有设计稿，做可信还原
2. 无设计稿，做保守补设计

它不应该自动演化成一长串设计文档流程。

## 适用场景

- Figma/截图/标注稿还原
- 需求已明确，但缺设计稿，需要前端补设计
- 设计系统对齐

## 路径判定

### 路径 A：有设计稿

重点是：

- 解析设计
- 对照项目现有组件和 token
- 说明可复用项与偏差项
- 还原并验证

### 路径 B：无设计稿

重点是：

- 基于项目现有风格做保守设计
- 只在存在明显分叉时给出多方案
- 不要脱离现有产品语言和设计系统自由发挥

## 硬规则

1. 先读项目已有页面和组件，不凭空设计。
2. 有设计稿时，先看复用和还原，不先讨论技术流程。
3. 无设计稿时，优先贴合项目现有风格，而不是“做更漂亮的东西”。
4. 不要求所有设计任务都先初始化或先建 Item。

## 风险分级建议

| 级别 | 特征 | 默认策略 |
|------|------|----------|
| L1 | 局部设计落地或小范围补设计 | 直接实现 + 简短偏差说明 |
| L2 | 多组件、多端适配或存在设计分叉 | 短方案 + 还原/实现/走查 |
| L3 | 设计系统扩展、共享组件、全局视觉变更 | 强治理 + 明确设计约束 |

## 实施重点

### 有稿还原

至少说明：

- 设计的结构和关键视觉层次
- 现有系统中哪些能复用
- 哪些地方需要偏差处理
- 偏差是因为设计意图不明还是技术约束

### 无稿补设计

至少说明：

- 已知输入
- 设计目标
- 参考的现有页面或组件
- 为什么当前方案最贴近项目既有风格

## 验证重点

- 关键页面或组件是否符合预期
- 多状态表现是否合理
- 响应式和文本表现是否稳定
- 偏差是否已明确说明

## 何时需要确认

- 有多种设计方向，差异明显
- 需要突破现有设计系统
- 还原时存在不可避免偏差，影响体验或品牌一致性

其余情况不需要把设计任务拆成多轮审批。
