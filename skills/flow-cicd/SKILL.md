# flow-cicd — CI/CD 变更流程

当 orchestrator 将任务分类为 CICD 时，按本流程推进。

核心原则：**改的是交付系统，没有回滚方案就不该上。**

## 前置条件

- feflow 已初始化，memory-load 已执行，Item 已生成（如 `CICD-001`）
- 分支命名：`cicd/{ITEM-ID}-{slug}`

**产出文件路径约定**：所有产出存放在 `.feflow/items/{ITEM-ID}/` 下。

---

## 阶段 1：现状调研（不可跳过）

1. 梳理当前流水线全貌：触发条件、各步骤职责、执行顺序
2. 标记环境依赖：secret 使用情况、凭证、环境变量
3. 标记**必须保留的步骤**和**不可碰的依赖**
4. 确认发布触发机制：手动 / tag / merge / schedule
5. 产出 `cicd-baseline.md`

---

## 阶段 2：变更目标

明确要改什么、为什么改、改完预期效果。典型场景：加缓存/并行化/提速、preview deploy、lint gate、部署策略调整、环境变量/secret 调整等。

**禁止模糊目标和"顺手优化"。**

---

## 阶段 3：风险评估

| 风险维度 | 检查内容 |
|---------|---------|
| 部署稳定性 | 变更是否影响正常部署流程 |
| Secret 安全 | 是否涉及 secret 增删改、是否暴露敏感信息 |
| 回滚能力 | 变更后能否回到上一个可用状态 |
| 多环境一致性 | dev / staging / prod 是否都能正常工作 |
| 依赖链 | 上下游流水线是否受影响 |

**硬规则：任何一项风险无法回答"可控"，必须先解决再继续。**

---

## 阶段 4：变更计划

1. 分步描述改动内容和顺序
2. 每步包含：改什么、预期结果、验证方式、**回滚方式**
3. 产出 `cicd-plan.md`

---

## 阶段 5：计划评审

**必须经用户确认。未确认不可实施。**

- **通过** → 进入阶段 6
- **修改** → 回到阶段 4
- **退回** → 回到阶段 3

---

## 阶段 6：灰度验证

1. 优先在非生产环境验证（dev / staging）
2. 验证链路：流水线触发 → 各步骤执行 → 产物正确 → 部署成功
3. 建议调用 QA agent 验证
4. 产出 `role-qa/cicd-test-report.md`

---

## 阶段 7：正式切换

1. 创建 PR（标题：`ci({scope}): {描述} [{ITEM-ID}]`）
2. 合并后监控首次执行，确认全流程正常
3. 异常立即执行回滚预案

---

## 阶段 8：回滚预案验证

1. 回滚方案必须在正式切换**前**准备完毕
2. 切换后验证回滚方案仍可用（实际执行或 dry-run）
3. 产出 `rollback-note.md`

---

## 阶段 9：收尾

调用 memory-update 记录：变更内容、回滚方式、验证结果、踩坑点。更新 Item 状态为 `completed`。

---

## 状态流转

```
cicd_baseline → risk_assessed → plan_approved → verifying → switched → rollback_verified → completed
```

## 硬规则汇总

1. **没有回滚方案就不该上** — 每个改动步骤必须有回滚路径
2. **灰度先行** — 非生产环境验证通过才能切换生产
3. **风险不可控就停** — 任何风险项无法确认可控，阻断流程

## 异常处理

- **灰度验证失败**：回到阶段 4 修订计划，不强行推进
- **正式切换后异常**：立即执行回滚预案，回到阶段 3 重新评估
- **回滚预案不可用**：阻断流程，通知用户，优先修复回滚能力
