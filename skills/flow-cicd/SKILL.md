# flow-cicd — v2 CI/CD 变更流程

`flow-cicd` 是高风险 Delivery 的专用流程。

核心原则：

1. 改的是交付系统，不是普通业务代码
2. 没有回滚意识就不该推进
3. 灰度和验证优先于文档完整性

## 适用场景

- pipeline 调整
- 部署策略变更
- preview/release 流程修改
- 环境变量或 secret 使用方式变更
- 构建提速或缓存策略改造

## 硬规则

1. 必须先理解当前流水线和依赖关系。
2. 必须知道失败后怎么回退。
3. 必须优先在非生产环境验证。
4. 风险不清楚时必须停下来确认。

## 默认节奏

1. 梳理现状
2. 说明要改什么、为什么改
3. 列出风险与回滚路径
4. 在安全环境验证
5. 再切到正式环境

## 风险关注点

- 部署稳定性
- secret / env 安全
- 多环境一致性
- 上下游依赖
- 回滚可行性

## 何时需要确认

- 回滚路径不明确
- 会触及生产环境关键链路
- 会改变发布触发机制或审批机制
- 会影响多个环境或团队

## 验证要求

至少验证：

- 流程能触发
- 关键步骤能执行
- 产物和部署结果正确
- 回滚路径可用

这类任务默认应视作 L3 对待。
