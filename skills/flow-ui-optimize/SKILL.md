# flow-ui-optimize — UI/UX 优化流程

当 orchestrator 将任务分类为 UI 时，按本流程推进。

核心原则：**方案评审重点是视觉和交互，不是技术方案。** 先确认"改成什么样"，再讨论"怎么改"。

## 前置条件

- feflow 已初始化（`.feflow/` 目录存在且 `init-config.md` 配置完整）
- orchestrator 已完成任务分类并生成 Item（如 `UI-001`）
- memory-load 已执行

复杂度等级定义同 flow-feature。分支命名：`ui/{ITEM-ID}-{slug}`。
**产出文件路径约定**：所有产出文件存放在 `.feflow/items/{ITEM-ID}/` 下。

---

## 阶段 1：问题定义（不是需求）

**目标**：明确当前 UI/UX 问题是什么，想改善什么。注意：这里是"问题"而非"需求"。

1. **强制代码调研（不可跳过）**：用 Grep/Glob 搜索涉及的组件/页面/样式文件，用 Read 读取现有实现。**声称"不存在"时必须展示搜索证据**（关键词、目录、结果），否则视为流程违规。
2. 收集问题证据：截图、用户反馈、可用性数据
3. 分类问题类型：视觉不一致 / 交互不合理 / 响应式缺陷 / 动效缺失 / 可用性障碍 / 无障碍不达标
4. 产出：`role-pm/problem-definition.md`

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 问题分析 | 自行分析 | PM agent | PM + designer agent |
| 证据要求 | 文字描述 | 附截图/对比 | 附数据+用户反馈 |

---

## 阶段 2：视觉与交互方案（本流程核心，不可跳过）

**必须调用 designer agent。** 产出 `role-designer/ui-proposal.md`（使用模板 `templates/ui-proposal.md`），包含：当前问题（附截图）、改善目标、参考风格、视觉调整（布局/色彩/字体/间距/层级）、交互调整（用户路径/操作点/状态变化/异常态覆盖）、影响范围、适配要求（响应式/暗黑/i18n/无障碍）。

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 方案深度 | 直接给出方案 | 方案+对比 | 方案 A/B + 推荐理由 |
| designer agent | 建议调用 | 必须调用 | 必须调用+设计系统审查 |
| 适配范围 | 按需 | 响应式必查 | 响应式+暗黑+i18n+无障碍 |

---

## 阶段 3：方案评审

**所有级别必须经用户确认。评审视觉和交互方案，不评审技术实现。AI 不可自行跳过评审。**

呈现 `ui-proposal.md` 给用户确认：优化方向、视觉风格、交互逻辑、异常态覆盖。
通过 → 阶段 4 / 修改 → 回阶段 2 / 退回 → 回阶段 1。

---

## 阶段 4：开发计划

1. L1 自行制定；L2 调用 FE agent；L3 调用 FE + designer agent
2. **深度代码调研（不可跳过）**：搜索相关组件/样式代码，确认设计系统 token 复用范围，引用文件路径必须经 Glob/Read 验证
3. 优先复用设计系统已有 token/组件，标注需新建项
4. 产出：`role-fe/dev-plan.md`

---

## 阶段 5：实施

1. **门禁检查**：调用 quality-gate skill 确认前置条件
2. **实施顺序**：先改样式/结构，后改逻辑；优先复用设计系统 token/组件
3. 持续记录 `implementation-log.md`

---

## 阶段 6：测试验证（必须覆盖 UI 状态完整性检查）

1. **UI 状态完整性检查（强制，不可跳过）**：调用 designer agent 检查清单逐项确认——loading / empty / error / success / disabled / hover+focus / mobile / dark / i18n 溢出 / 登录态 / 权限态。未覆盖项必须补充方案或标注"不适用"+原因。
2. 视觉回归检查：多端 / 多主题 / 多状态截图对比
3. 运行测试套件（lint / type-check / unit test）
4. 产出：`role-qa/test-report.md`

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 状态检查 | 核心状态 | 全部状态 | 全部状态+边界场景 |
| 视觉回归 | 主端确认 | 主端+移动端 | 全端+暗黑+i18n |

---

## 阶段 7：收尾

1. 创建 PR（标题：`style({scope}): {描述} [{ITEM-ID}]`）
2. 调用 memory-update skill 写入项目记忆，记录：UI 变更点、设计系统影响、踩坑点
3. 更新 Item 状态为 `completed`

---

## 状态流转

```
problem_defined → ui_proposal_draft → ui_proposal_approved → plan_draft → implementing → testing → completed
```

任意阶段可退回前置阶段，退回原因记录在 `implementation-log.md`。

## 异常处理

- **设计系统冲突**：优先遵循设计系统，若需突破则记录原因并同步维护者
- **多端表现不一致**：逐端确认方案，不假设"自适应就行"
- **UI 状态检查未覆盖**：补充方案后重新评审，不跳过
- **用户未响应评审**：提醒一次后等待，不自行推进
