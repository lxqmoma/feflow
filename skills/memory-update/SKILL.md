# memory-update: 任务后经验提炼

## 触发时机

由 orchestrator 在任务完成后、Item 状态流转为 `done` 时调用。
目的是从本次任务中提炼值得沉淀的经验，写入记忆库供后续任务复用。

## 输入

orchestrator 传入当前 Item 的完整信息：

- `item_id` — 工作项标识（如 `FEAT-0012`）
- `item_type` — 类型（feat / fix / refactor / hotfix / chore）
- `modules` — 涉及模块列表（如 `["auth", "payment"]`）
- `risk_level` — 风险级别（low / medium / high / critical）
- `title` — 任务标题
- `description` — 任务描述
- `evidence` — 任务产出的证据清单（测试结果、审查记录等）
- `duration` — 实际耗时
- `review_comments` — 代码审查意见（若有）
- `rework_count` — 返工次数

## 评估清单

对本次任务进行以下 7 项自查，判断是否产生了值得沉淀的经验：

### 1. 新坑发现

本次任务是否遇到了之前未记录的坑？

判定条件：
- 任务执行过程中遇到非预期的阻塞
- 发现了文档中未提及的隐含约束
- 某个 API 或依赖的实际行为与预期不符

### 2. 复发问题

本次遇到的问题是否在记忆库中已有记录但仍然发生？

判定条件：
- `00-memory-brief.md` 中已提及该问题
- 执行过程中仍然踩到了相同的坑
- 说明现有规避方式不够有效，需要升级处理

### 3. 评审反馈

代码审查中是否收到了具有通用价值的反馈？

判定条件：
- 审查意见指出的问题不仅适用于本次代码，而是一类通用模式
- 审查意见涉及架构设计、安全规范、性能要求等全局性问题
- 审查意见纠正了对某个 API 或框架的错误用法

### 4. 返工发生

本次任务是否发生了返工？

判定条件：
- `rework_count > 0`
- 返工原因为需求理解偏差、技术方案不合理或遗漏边界条件
- 非因需求变更导致的返工

### 5. 方案否决

初始方案是否被否决，最终采用了不同方案？

判定条件：
- 初始技术方案在评审或实施中被否定
- 最终方案与初始方案有实质性差异
- 否决原因具有参考价值（非纯偏好差异）

### 6. 设计偏差

实际实现是否与设计文档存在偏差？

判定条件：
- 实现过程中发现设计文档的假设不成立
- 因技术限制调整了接口设计或数据结构
- 偏差需要回溯更新设计文档

### 7. 流程绕过

本次任务是否绕过了标准流程的某个环节？

判定条件：
- 因时间紧迫跳过了测试、审查或其他质量检查
- 使用了临时方案绕过了已知限制
- 绕过行为需要在后续补齐

## 评估结果判定

- 7 项全部为否 — 无新经验，不写入记忆库
- 任意一项为是 — 进入升级策略判定，确定写入层级

## 升级策略

根据经验的严重程度和出现频次，分四个层级处理：

### Level 1 — 首次发现，不升级

**条件**：新坑首次出现，影响范围有限，不涉及安全和数据完整性。

**处理方式**：
- 不写入记忆库的独立文件
- 仅在当前 Item 的完成记录中留痕
- 在 Item 的 `lessons_learned` 字段中记录

**理由**：首次出现的问题可能是偶发的，过早写入记忆库会造成噪音。如果它再次出现，Level 2 会捕获它。

### Level 2 — 第二次出现，写入模块记忆

**条件**：相同或相似问题第二次出现（通过与历史 Item 的 `lessons_learned` 比对判定）。

**处理方式**：
- 写入 `.feflow/memory/modules/{module_name}.md`
- 若文件不存在则创建
- 追加新的记忆条目，不修改已有条目
- 同时将相关 Item 的 `lessons_learned` 标记为 `escalated`

**写入位置**：
- 若问题仅涉及单一模块，写入对应模块文件
- 若问题跨模块，写入 `.feflow/memory/patterns/` 目录下的独立文件

### Level 3 — 高风险问题，升格为不变量

**条件**（满足任一即触发）：
- 问题涉及安全、数据完整性或资金相关逻辑
- 问题曾导致线上事故（`type: incident`）
- 问题的修复成本极高（需要回滚发布、修复数据等）
- `risk_level` 为 critical 的任务中发现的问题

**处理方式**：
- 写入 `.feflow/memory/project/invariants.md`
- 追加新的不变量条目
- 同时在对应模块记忆中添加引用
- 标记 `priority: critical`

### Level 4 — 反复复发，建议补充自动化防护

**条件**：同一问题出现三次及以上，说明靠记忆提醒不足以防止复发。

**处理方式**：
- 更新对应记忆条目的 `status` 为 `needs_automation`
- 在产出中给出具体建议：
  - 补充自动化测试用例覆盖该场景
  - 添加 lint 规则或 pre-commit 检查
  - 添加 CI 流水线检查点
  - 更新代码模板或脚手架
- 记忆条目中记录建议的自动化方案

## 记忆写入格式

每条记忆在目标文件中以如下格式追加：

```markdown
---
memory_id: MEM-A012
type: pitfall
scope:
  - auth
created_at: 2026-04-08T15:30:00+08:00
status: active
source_type: review_feedback
source_refs:
  - FEAT-0012
created_by: memory-update
---

### [MEM-A012] JWT 刷新接口的并发安全问题

**场景**：多个前端请求在 token 即将过期时并发发起刷新。

**问题**：刷新接口未做并发控制，多个请求各自拿到不同的新 token，导致部分请求携带的 token 被作废。

**根因**：`refreshToken()` 函数缺少锁机制，每次调用都会发起独立的刷新请求。

**规避方式**：引入请求级别的锁或队列机制，确保同一时刻仅有一个刷新请求在进行中，其余调用方等待该请求完成后共用结果。

**来源 Item**：FEAT-0012（auth 模块登录态优化）

**记录时间**：2026-04-08
```

### 记忆条目 frontmatter 元数据说明

每条记忆必须包含以下 frontmatter 字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `memory_id` | string | 是 | 唯一标识，格式为 `MEM-{模块首字母大写}{序号}`，如 `MEM-A012` |
| `type` | enum | 是 | 记忆类型：`pitfall`(踩坑) / `pattern`(模式) / `incident`(事故) / `convention`(约定) / `invariant`(不变量) |
| `scope` | string[] | 是 | 适用范围，模块名列表，跨模块问题用 `["global"]` |
| `created_at` | datetime | 是 | 创建时间，ISO 8601 格式，带时区 |
| `status` | enum | 是 | 状态：`active`(生效中) / `archived`(已归档) / `superseded`(已被替代) / `needs_automation`(需自动化) |
| `source_type` | enum | 是 | 来源类型：`new_discovery`(新发现) / `recurrence`(复发) / `review_feedback`(评审反馈) / `rework`(返工) / `plan_rejection`(方案否决) / `design_deviation`(设计偏差) / `process_bypass`(流程绕过) |
| `source_refs` | string[] | 是 | 关联的 Item ID 列表 |
| `created_by` | string | 是 | 固定值 `memory-update`，标识由本 skill 自动创建 |

### memory_id 编号规则

- 前缀 `MEM-`
- 模块标识：取模块名首字母大写（如 auth -> A, payment -> P, global -> G）
- 序号：三位数字，从 001 开始递增
- 生成新 ID 时，读取目标文件中已有的最大序号，+1 生成新序号
- 若目标文件不存在，从 001 开始

## 输出格式

### 有新经验需要沉淀时

```markdown
## memory-update 执行报告

### 新增记忆

| memory_id | 类型 | 范围 | 层级 | 写入位置 |
|-----------|------|------|------|----------|
| MEM-A012 | pitfall | auth | Level 2 | `.feflow/memory/modules/auth.md` |
| MEM-P008 | pattern | payment | Level 3 | `.feflow/memory/project/invariants.md` |

### 记忆详情

#### MEM-A012 — JWT 刷新接口的并发安全问题
- 来源：FEAT-0012 代码审查反馈
- 层级：Level 2（第二次出现，写入模块记忆）
- 规避方式：引入请求锁机制

#### MEM-P008 — 支付回调签名验证必须在依赖升级后回归
- 来源：FEAT-0012 实施过程中发现
- 层级：Level 3（涉及资金安全，升格为不变量）
- 规避方式：依赖升级 CI 流水线中增加支付回归测试步骤

### 建议后续操作

1. MEM-A012：建议在 auth 模块中补充并发刷新的单元测试
2. MEM-P008：建议在 CI 配置中增加依赖升级后的支付回归检查
```

### 无新经验时

```markdown
## memory-update 执行报告

本次任务未产生需要沉淀的新经验。

评估结果：
- 新坑发现：否
- 复发问题：否
- 评审反馈：无通用价值的新反馈
- 返工发生：否
- 方案否决：否
- 设计偏差：否
- 流程绕过：否

无需更新记忆库。
```

## 执行约束

1. memory-update 只追加内容，不修改或删除已有的记忆条目
2. 写入前必须读取目标文件已有内容，避免覆盖
3. 若目标文件不存在，创建文件并写入标准文件头（文件级 frontmatter + 说明）
4. 每次执行最多写入 5 条新记忆，超出时按优先级保留：
   - Level 3（不变量）优先级最高
   - Level 4（需自动化）次之
   - Level 2（模块记忆）再次
5. 写入的记忆条目必须具备可操作性——"注意 XXX"不是有效的规避方式，需具体到做什么、怎么做
6. 不对评估清单中的判定做虚假肯定——不确定是否构成经验时，宁可不写入

## 与其他 skill 的关系

- **上游**：orchestrator 在 Item 完成后调用本 skill
- **下游**：写入的记忆供后续任务的 memory-load 检索
- **对称**：memory-load 在任务前读取记忆，memory-update 在任务后写入记忆，构成完整的经验循环
- **关联**：quality-gate 的检查结果可作为本 skill 评估清单的输入依据
