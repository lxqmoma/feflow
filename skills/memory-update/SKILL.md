# memory-update: v2 任务后经验沉淀

## 触发时机

建议在任务完成后按需调用。

不是每个任务结束都必须调用。

## 输入

推荐输入：

`item_id` / `task_type` / `modules` / `risk_level` / `title` / `evidence` / `rework_count` / `incident_flag`

## 进入判定

只有在本次任务产生了真正可复用信息时，才考虑写入记忆。

高价值信号包括：

1. 发现了新约束或新坑
2. 历史问题再次复发
3. 形成了值得复用的新模式
4. 发生了事故、回滚或高风险返工
5. 修正了一个长期错误认知

若没有这些信号，默认不写记忆。

## 写入层级

| Level | 条件 | 处理 | 写入位置 |
|-------|------|------|----------|
| 1 | 影响有限，但值得留痕 | 可只留在 Item 或实施记录中 | Item / implementation log |
| 2 | 与模块强相关、未来高概率再遇到 | 写入模块记忆 | `.feflow/memory/modules/` |
| 3 | 跨模块通用模式或高频陷阱 | 写入 patterns | `.feflow/memory/patterns/` |
| 4 | 涉及事故、权限、资金、数据一致性等高风险约束 | 写入 incidents 或 invariants | `.feflow/memory/incidents/` / `project/invariants.md` |

## 记忆内容要求

一条有效记忆至少应回答：

1. 场景是什么
2. 问题或约束是什么
3. 为什么会发生
4. 以后该怎么避免

“注意 XXX” 这种没有操作性的句子不应入库。

## 输出格式

有新经验时输出：

- 写入了什么
- 为什么值得写
- 写到了哪一层
- 是否还需要 follow-up

无新经验时直接返回“无需更新记忆”即可，不必冗长解释。

## 执行约束

1. 只追加不修改已有记忆条目
2. 写入前必须读取目标文件已有内容，避免覆盖
3. 目标文件不存在时创建（含标准文件头）
4. 每次最多写入 5 条
5. 不确定是否值得沉淀时，宁可不写
6. 不要把普通小改动、一次性文案修改、低价值过程信息写入长期记忆

## 与其他 skill 的关系

- `memory-load` 负责取回高价值历史信息
- `memory-update` 负责沉淀新的高价值信息
- 两者是闭环，但都不应被机械地对每个任务执行
