# flow-feature — 新功能开发流程

当 orchestrator 将任务分类为 FEAT（V1 中也复用于 MOD / UI / DESIGN / CHANGE / REFACTOR / TEST / CICD / DEBT）时，按本流程推进。

## 前置条件

- feflow 已初始化（`.feflow/` 目录存在且 `init-config.md` 配置完整）
- orchestrator 已完成任务分类并生成 Item（包含唯一 ID，如 `FEAT-001`）
- memory-load 已执行，当前会话拥有项目记忆上下文

## 复杂度等级定义

| 等级 | 判断标准 | 典型场景 |
|------|---------|---------|
| L1 | 单文件修改，逻辑清晰，无跨模块影响 | 修改文案、调整样式、添加简单工具函数 |
| L2 | 多文件联动，涉及组件/接口/状态变更 | 新增页面、对接新接口、重构组件结构 |
| L3 | 跨模块/跨系统，涉及架构决策或重大变更 | 引入新框架、全局状态重构、SSR 改造 |

---

## 阶段 1：需求理解

**目标**：将用户输入转化为结构化的需求简报。

**执行逻辑**：

1. 读取用户输入（自然语言描述、截图、链接等）
2. 调用 memory-load 获取项目上下文（技术栈、目录结构、已知约定）
3. **强制代码调研（不可跳过，必须展示搜索证据）**：
   - 从用户描述中提取所有核心名词（组件名、功能名、模块名、技术术语）
   - 用 Grep 搜索**整个项目**（包括 src/、packages/、apps/ 等所有子目录），展示搜索结果
   - 用 Glob 扫描 `**/*.vue`、`**/*.ts`、`**/*.tsx` 找到所有相关文件
   - 对搜索到的文件用 Read 读取，理解现有实现
   - **声称"没有xxx"时必须展示完整搜索证据**：搜了哪些关键词、搜了哪些目录、结果是什么
   - 未展示搜索证据就说"不存在"，视为流程违规
4. 根据复杂度等级执行分析：
   - L1：自行分析，直接产出需求简报
   - L2/L3：调用 PM agent，进行多角度需求拆解

5. 产出文件：`.feflow/items/{ITEM-ID}/role-pm/requirement-brief.md`

**需求简报模板**（产出至 `role-pm/requirement-brief.md`）：

```markdown
# 需求简报 — {ITEM-ID}
## 基本信息
- Item ID / 类型 / 复杂度 / 创建时间
## 需求描述
{用一段话概括要做什么以及为什么}
## 预期行为
- {完成后系统应表现出的行为，条目式}
## 验收标准
- [ ] {可检验的条件}
## 影响范围
- 涉及模块 / 涉及文件（预估） / 依赖变更（有/无）
## 约束与风险
## 关联上下文
- 相关历史 Item / 相关记忆条目
```

**L1/L2/L3 差异**：

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 分析方式 | 自行分析 | 调用 PM agent | 调用 PM agent + architect agent |
| 影响范围 | 单文件预估 | 多文件列举 | 跨模块依赖分析 |
| 验收标准 | 1-2 条 | 3-5 条 | 5+ 条，含非功能性要求 |
| 风险分析 | 可省略 | 必须列出 | 必须列出 + 缓解方案 |

---

## 阶段 2：需求评审

**所有级别都必须经过用户确认才能进入下一阶段。AI 不可自行判断"这个简单"就跳过评审。**

将需求简报的核心内容呈现给用户，等待明确反馈：
- **通过** → status 更新为 `requirement_approved`，进入阶段 3
- **补充** → 修订后重新呈现
- **退回** → status 更新为 `requirement_rejected`，回到阶段 1

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 呈现详细度 | 一句话确认 | 摘要 + 验收标准 | 全文 + 架构影响说明 |

**L1 的确认方式**：至少问一句"我理解的是 xxx，这样对吗？确认后我开始做。"

**硬规则：AI 不可在流程中途自行降级任务级别来绕过评审。** 初始级别由 orchestrator 确定后，不允许 flow skill 自行改变。

---

## 阶段 3：开发计划

**目标**：制定可执行的开发计划。

**执行逻辑**：

1. 调用 FE agent（或 L1 时自行制定）
2. **深度代码调研（不可跳过）**：
   - 搜索并阅读与本任务直接相关的现有代码（组件、模块、工具函数、样式）
   - 理解现有实现的结构、依赖关系和调用链
   - 确认哪些能复用、哪些要修改、哪些要新建
   - 开发计划中引用的文件路径必须经过实际验证（Glob/Read 确认存在）
3. 基于 `requirement-brief.md`、项目记忆和代码调研结果，产出开发计划
4. **必须包含"历史问题对照"段落**：查询 memory 中与当前任务相关的历史问题、踩坑记录、已知限制，在计划中明确标注并给出规避措施
4. 产出文件：`.feflow/items/{ITEM-ID}/role-fe/dev-plan.md`

**开发计划结构**（产出至 `role-fe/dev-plan.md`）：

```markdown
# 开发计划 — {ITEM-ID}
## 实施步骤
1. {步骤描述} — 涉及文件 / 预估改动量（小/中/大）
## 技术方案
## 历史问题对照
| 历史问题 | 来源 | 与本次的关联 | 规避措施 |
|---------|------|------------|---------|
## 测试策略（单元/集成/手动）
## 分支策略
- 分支名: feature/{ITEM-ID}-{slug}，基于: {base branch}
```

**L1/L2/L3 差异**：

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 制定方式 | 自行制定 | 调用 FE agent | 调用 FE agent + architect agent 联合 |
| 步骤粒度 | 1-3 步 | 3-8 步 | 8+ 步，按阶段拆分 |
| 历史问题对照 | 快速扫描 | 必须列出 | 深度关联分析 |
| 测试策略 | 可简化 | 必须包含 | 必须包含 + 性能/兼容性考量 |

---

## 阶段 4：计划评审

**所有级别都必须经过用户确认才能进入实施。**

将开发计划的核心内容呈现给用户，等待反馈：
- **通过** → status 更新为 `plan_approved`，进入阶段 5
- **修改** → 调整后重新呈现
- **退回** → 回到阶段 3 重新制定

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 呈现详细度 | "我打算改 xxx 文件的 xxx 行，可以吗？" | 步骤合理性 | 架构影响 + 逐项确认关键决策 |

**禁止未经用户确认就开始写代码。**

---

## 阶段 5：实施

**目标**：按计划完成代码实现。

**执行逻辑**：

1. **门禁检查**：调用 quality-gate skill，确认进入实施阶段的前置条件已满足
2. **分支创建**：创建分支 `feature/{ITEM-ID}-{slug}`
3. **TDD 建议**：检测 superpowers 是否可用，若可用则建议使用 TDD 工作流（test-driven-development skill）
4. **编码实施**：
   - 遵循 AI 编码戒律（见下方）
   - 按 `dev-plan.md` 的步骤顺序推进
   - 每完成一个步骤，更新 implementation-log
5. **持续记录**：产出并持续更新 `.feflow/items/{ITEM-ID}/implementation-log.md`

**AI 编码戒律**（实施阶段必须遵守）：

- 优先做最小必要修改，不扩大改动范围
- 不顺手重构无关代码
- 不引入计划外依赖
- 不修改不相关的文件
- 修改前先理解调用链和影响面
- 优先定位根因，不做表面补丁
- 代码注释使用中文，只在必要时添加
- 变量 camelCase，常量 UPPER_SNAKE_CASE，类型/组件 PascalCase

**implementation-log.md 结构**（持续更新）：

```markdown
# 实施日志 — {ITEM-ID}
## 步骤进度
| 步骤 | 状态 | 备注 |
|------|------|------|
## 关键决策记录
## 遇到的问题
```

**L1/L2/L3 差异**：

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 门禁检查 | 基础检查 | 完整检查 | 完整检查 + 架构合规 |
| TDD | 可选 | 建议使用 | 强烈建议使用 |
| 实施日志 | 简要记录 | 按步骤记录 | 按步骤记录 + 决策依据 |
| 代码审查 | 可省略 | 建议 code-review | 必须 code-review |

---

## 阶段 6：测试验证

**目标**：验证实现符合需求，且未引入回归问题。

**执行逻辑**：

1. 调用 QA agent 执行测试验证
2. 运行已有测试套件（lint / type-check / unit test / e2e）
3. 针对验收标准逐项验证
4. **历史回归检查**：查询 memory 中相关模块的历史 bug 和修复记录，确认未回归
5. 产出文件：`.feflow/items/{ITEM-ID}/role-qa/test-report.md`

**测试报告结构**（产出至 `role-qa/test-report.md`）：

```markdown
# 测试报告 — {ITEM-ID}
## 验收标准验证
| 验收标准 | 结果 | 证据 |
|---------|------|------|
## 自动化测试
- lint / type-check / unit test / e2e（如适用）
## 历史回归检查
| 历史问题 | 来源 | 回归状态 | 验证方式 |
|---------|------|---------|---------|
## 结论
- 测试结果 / 未通过项
```

**L1/L2/L3 差异**：

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 执行方式 | 自行验证 | 调用 QA agent | 调用 QA agent |
| 测试范围 | lint + 手动验证 | lint + type-check + unit test | 全部 + e2e + 性能 |
| 历史回归 | 快速扫描 | 必须检查 | 深度检查 + 相邻模块 |
| 报告详细度 | 简要 | 完整报告 | 完整报告 + 性能基准 |

---

## 阶段 7：发布

**目标**：将代码合并至目标分支并产出发布记录。

**执行逻辑**：

1. 确认阶段 6 测试全部通过
2. 创建 PR：
   - PR 标题格式：`feat({scope}): {描述} [{ITEM-ID}]`
   - PR 描述包含：需求摘要、实施概要、测试结果、影响范围
3. 产出文件：`.feflow/items/{ITEM-ID}/release-note.md`

**release-note.md 结构**：

```markdown
# 发布记录 — {ITEM-ID}
## 变更概述（一段话描述内容和价值）
## 变更类型 / 影响范围
## 变更明细
## 测试验证（状态 + 报告位置）
## 注意事项
```

**L1/L2/L3 差异**：

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| PR 创建 | 可直接合并（无 PR） | 必须创建 PR | 必须创建 PR + 指定 reviewer |
| release-note | 简要 | 完整 | 完整 + 迁移指南（如需） |
| 合并方式 | 直接合并 | squash merge | 视情况选择 |

---

## 阶段 8：收尾

调用 memory-update skill，将关键信息写入项目记忆（新增/变更的模块和接口、关键决策、踩坑点、可复用模式）。更新 Item 状态为 `completed`，清理临时文件。

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 记忆更新 | 简要记录 | 完整记录 | 完整记录 + 架构决策归档 |
| 经验沉淀 | 可省略 | 记录踩坑点 | 记录踩坑点 + 最佳实践 |

---

## 状态流转总览

```
requirement_draft
  → requirement_approved (阶段2通过)
  → plan_draft
  → plan_approved (阶段4通过)
  → implementing
  → testing
  → releasing
  → completed
```

任意阶段发现阻塞问题时，可退回到前置阶段。退回操作会记录在 implementation-log.md 中，包含退回原因。

## 异常处理

- **用户长时间未响应评审**：L2 提醒一次后等待，L3 必须等待明确确认
- **实施中发现需求理解偏差**：暂停实施，回到阶段 1 修正需求简报
- **测试不通过**：回到阶段 5 修复，修复后重新执行阶段 6
- **依赖阻塞**：记录阻塞原因至 implementation-log.md，通知用户
