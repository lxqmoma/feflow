# flow-modification — v2 旧功能修改流程

`flow-modification` 是 `flow-feature` 的特化版本。

它的独特要求只有一个：

**修改旧功能时，最重要的不是“怎么改”，而是“哪些现有行为不能被误伤”。**

## 适用场景

- 旧功能行为调整
- 兼容逻辑修改
- 页面、组件、状态或接口的增量修改
- 需求不完全是新增，而是建立在现有行为上的调整

## 与通用 Delivery 的区别

沿用 `flow-feature` 的风险驱动节奏，但要额外做两件事：

1. 建立现状行为基线
2. 在实施和验证时持续对照基线

## 硬规则

1. 先读现有实现，再定义改动目标。
2. 不要只描述“想变成什么样”，还要说明“现在是什么样”。
3. 不要因为是修改任务就默认走完整评审流水线。
4. L1 修改允许直接做，但必须先确认关键行为基线。

## 现状行为基线

在开始实施前，至少搞清楚：

- 当前正常路径
- 当前异常路径
- 依赖的共享状态或接口契约
- 兼容逻辑、临时逻辑、历史 hack
- 哪些行为绝对不能被破坏

L1 可用内联摘要完成。
L2/L3 如需追踪，可沉淀为简短基线记录。

## 风险分级建议

| 级别 | 特征 | 默认策略 |
|------|------|----------|
| L1 | 局部行为微调 | 建立最小基线后直接改 |
| L2 | 多文件或多状态面修改 | 先做简短基线和影响分析 |
| L3 | 共享契约、兼容逻辑、跨模块修改 | 启用重治理与回滚意识 |

## 实施要求

### L1

- 读相关文件
- 记录最小行为基线
- 实施修改
- 验证旧行为未被误伤，新行为符合目标

### L2

- 先输出“当前行为 vs 目标行为”摘要
- 明确哪些行为保持不变
- 实施后重点做回归验证

### L3

- 明确影响面和兼容边界
- 若 `.feflow/` 存在，应追踪关键决策
- 必须有回滚意识和回归验证策略

## 验证重点

修改任务里，回归优先级通常高于新增验证。

至少验证：

1. 目标行为确实变化了
2. 非目标行为没有明显退化
3. 兼容边界没有被破坏

## 何时需要确认

- 目标行为本身有歧义
- 修改会破坏既有兼容性
- 两种方案对已有用户体验影响不同

除此之外，不要机械停下来做“评审确认”。
