# flow-modification — 旧功能修改流程

当 orchestrator 将任务分类为 MOD 时，按本流程推进。

核心原则：**修改旧功能最大风险不是"做不出来"，而是"改炸了"。** 必须先建立现状行为基线，理解现有行为后再改动。

## 前置条件

- feflow 已初始化（`.feflow/` 目录存在且 `init-config.md` 配置完整）
- orchestrator 已完成任务分类并生成 Item（如 `MOD-001`）
- memory-load 已执行

复杂度等级定义同 flow-feature。分支命名：`modify/{ITEM-ID}-{slug}`。
**产出文件路径约定**：所有产出文件存放在 `.feflow/items/{ITEM-ID}/` 下。

---

## 阶段 0：现状行为基线分析（本流程独有，不可跳过）

**目标**：在任何修改之前，完整还原现有行为，明确哪些东西绝对不能动。

1. **行为调研（强制）**：用 Grep 搜索目标功能所有关键词，用 Glob 扫描相关文件，用 Read 读取代码。梳理完整行为（正常态 + 异常态），追踪数据流和调用链。**声称"不存在"时必须展示搜索证据**（关键词、目录、结果），否则视为流程违规。
2. **历史考古**：查询 memory 历史修改记录和踩坑点，检查 git log 修改频率，识别兼容逻辑（`// HACK` / `// FIXME` / `// 兼容`）
3. **约束清单**：列出不可破坏的行为、接口契约、向下兼容要求

**产出** `00-current-behavior-baseline.md`（使用模板 `templates/behavior-baseline.md`），包含：当前行为描述、相关依赖、历史兼容逻辑（逐条附代码位置）、不可破坏约束、已知技术债/风险点。

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 调研深度 | 目标文件+直接依赖 | 调用链+共享状态 | 全链路+跨模块影响 |
| 约束清单 | 1-2 条核心约束 | 3-5 条，含接口契约 | 完整约束+兼容性矩阵 |
| 历史考古 | 快速扫描 | 查 memory+git log | 深度关联+相邻模块 |

---

## 阶段 1：需求理解

参照 flow-feature 阶段 1，**关键差异**：需求描述必须包含"当前行为"与"期望行为"的对比，不接受只写期望行为。强制代码调研规则同 flow-feature。

---

## 阶段 2：需求评审

**所有级别必须经用户确认。AI 不可自行跳过评审或降级任务级别。**

呈现内容：当前行为 vs 期望行为的对比 + 基线中标注的不可破坏约束。
通过 → 阶段 3 / 补充 → 修订后重新呈现 / 退回 → 回到阶段 1。

---

## 阶段 3：开发计划

参照 flow-feature 阶段 3，**关键差异**：
1. 每步必须标注对基线行为的影响（"行为变更"或"行为保持"）
2. 必须包含"基线测试固化"步骤 — 为现有行为补充测试用例，确保改动前测试通过
3. 深度代码调研不可跳过，引用文件路径必须经 Glob/Read 验证

---

## 阶段 4：计划评审

**所有级别必须经用户确认才能进入实施。禁止未确认就写代码。**

重点审查：每步的行为影响标注是否准确、基线测试覆盖是否充分。
通过 → 阶段 5 / 修改 → 调整后重新呈现 / 退回 → 回到阶段 3。

---

## 阶段 5：实施

1. **门禁检查**：调用 quality-gate skill 确认前置条件
2. **基线测试固化（先于任何修改）**：为现有行为编写/补充测试，运行通过后再动代码
3. **编码实施**：按 `dev-plan.md` 顺序推进，每步完成后运行基线测试确认未回归
4. 持续记录 `implementation-log.md`

| 维度 | L1 | L2 | L3 |
|------|-----|-----|-----|
| 基线测试 | 核心路径 | 核心+边界 | 全覆盖 |
| 每步回归 | 手动确认 | 运行测试 | 运行测试+diff review |

---

## 阶段 6：测试验证

参照 flow-feature 阶段 6，**关键差异**：回归测试权重高于新功能验证。基线行为逐项回归，历史回归必查。

---

## 阶段 7：发布与收尾

参照 flow-feature 阶段 7-8。PR 标题：`fix({scope}): {描述} [{ITEM-ID}]`。memory-update 重点记录：行为变更点、基线偏移、踩坑点。

---

## 状态流转

```
baseline_analyzing → requirement_draft → requirement_approved → plan_draft → plan_approved → implementing → testing → releasing → completed
```

任意阶段可退回前置阶段，退回原因记录在 `implementation-log.md`。

## 异常处理

- **基线无法完整还原**：标注不确定部分，用户确认后再继续
- **修改影响超出预期**：暂停实施，回到阶段 0 重新评估基线
- **基线测试失败（改动前就挂）**：说明现有代码已有问题，与用户确认是否同步修复
- **需求理解偏差**：暂停实施，回到阶段 1
